# T23.B — 分工 + 协商流测试

## 你的任务

验证 Agent 分工推荐是否合理，成员接受/拒绝流程是否顺畅，拒绝后协商是否闭环，批量确认是否正确。

---

## 环境准备

### 1. 启动后端

```bash
cd backend
.venv\Scripts\Activate.ps1
python -m uvicorn app.main:app --reload --port 8000
```

### 2. 启动前端

新开一个终端：

```bash
cd frontend
npm install   # 首次需要
npm run dev
```

### 3. 加载种子数据

在浏览器访问 `http://localhost:8000/docs`，找到 `POST /api/seed/demo`，点击 "Try it out" → "Execute"。

这会加载当前完整 demo seed：
- 6 个用户（小林/小王/小张/小李/小赵/小刘）
- 1 个工作区 "ProjectFlow 团队"
- 1 个项目（active，已有方向卡）
- 4 个阶段（当前阶段为实现阶段，其余为 pending/completed 演示数据）
- 10 个任务（覆盖 P0/P1/P2、不同状态和负责人）
- 3 个已 finalized 的分工提案

种子数据已包含 finalized 分工，**你需要先重置再重新生成分工来测试完整流程**。

如果要严格验证“5 个任务 / 3 个成员 / 无 finalized 分工”的 T23.B 主路径，建议 reset 后通过测试脚本或 API 手动创建专用数据，不直接依赖完整 demo seed 的默认状态。

### 4. 替代方案：从 A 轨道接手

如果 A 轨道已完成，可以直接用 A 的项目继续测试。项目需已有阶段和任务，但**不应有 finalized 分工**。

### 5. 确认 LLM 配置

- **第一轮（mock 模式）**：`LLM_PROVIDER=mock`
- **第二轮（真实 LLM）**：`LLM_PROVIDER=openai`，`LLM_API_KEY=你的key`，`LLM_MODEL=gpt-4o-mini`

真实 LLM 第二轮执行前需确认可以消耗 API；不要把 API key 写入截图、日志或反馈文档。

### 6. 关于多成员操作

导航栏右侧有**用户切换下拉菜单**，可以切换当前身份为工作区内任意成员。

- 接受/拒绝其他成员的分工：先在导航栏切换到对应成员身份，再操作
- 批量确认阶段分工：切回项目创建者身份操作

## 前置条件确认

进入项目仪表盘后，确认以下状态：

- [ ] 项目状态为 active
- [ ] 有至少 1 个 active 阶段
- [ ] 有至少 3 个任务（P0/P1/P2 分布）
- [ ] 分工提案为空或全部为 proposed（无 finalized）
- [ ] "推荐下一步"显示"推荐分工"

如果使用种子数据但分工已 finalized，需要重置数据后只加载到任务拆解阶段。

## 测试步骤

### B1 — 运行分工推荐

1. 在项目仪表盘"Agent 操作"区域，找到"分工"阶段
2. 点击"推荐分工"按钮

**验证：**
- [ ] 按钮显示 loading（"运行中..."）
- [ ] 等待后，"分工流程"面板出现分工提案
- [ ] 每个提案包含：任务名、推荐负责人、备选人、理由
- [ ] 提案状态为"待确认"

**截图：** 分工流程面板 - 初始状态

---

### B2 — 查看分工提案详情

1. 逐个查看分工提案
2. 检查每个提案的详细信息

**验证：**
- [ ] 每个提案显示任务名（如"设计数据库 Schema"）
- [ ] 显示推荐负责人名字
- [ ] 显示备选人名字
- [ ] 显示理由（reason 字段）
- [ ] 可能有匹配度信息：技能匹配 / 时间匹配 / 偏好匹配 / 限制满足（应为中文标签，非英文 Skill/Availability/Preference/Constraint）
- [ ] 可能有风险提示（risk_note，红色背景）
- [ ] 有"接受分工"和"拒绝分工"两个按钮

**截图：** 单个提案展开详情

---

### B3 — 接受一个分工

1. 选择一个提案，点击"接受分工"按钮

**验证：**
- [ ] 提案状态变为"已确认"（Badge 颜色变化）
- [ ] 对应任务的 owner_user_id **尚未**更新——只有 finalized 后才写入 task.owner_user_id
- [ ] 分工面板或指标区"已确认待定"数量更新
- [ ] 仪表盘"分工覆盖率"仍按 finalized owner 计算，不因 owner_confirmed 提前变化
- [ ] 无页面崩溃

---

### B4 — 拒绝一个分工

1. 在导航栏用户切换下拉中，切换到被分配该任务的成员身份
2. 在分工流程面板中，找到该成员的提案，点击"拒绝分工"按钮
3. 出现拒绝表单：
   - 选择"偏好任务"（下拉选择其他任务）
   - 填写"原因"（如"我对后端更感兴趣"）
4. 点击"提交拒绝"

**验证：**
- [ ] 点击"拒绝分工"后，展开拒绝表单（有动画过渡）
- [ ] 偏好任务下拉列表包含所有任务
- [ ] 原因为必填项
- [ ] 提交后提案状态变为"已拒绝"（Badge 变红）
- [ ] 如果选择了偏好任务，协商面板出现新协商记录

**截图：** 拒绝表单展开状态

---

### B5 — 查看协商面板

1. 在分工流程面板底部，找到"协商面板"

**验证：**
- [ ] 协商面板显示协商记录
- [ ] 显示发起人（from_user_id 对应的成员名）
- [ ] 显示 Agent 消息（agent_message）
- [ ] 显示协商状态（pending/accepted/declined）
- [ ] 如果无协商记录，显示"暂无交换提议"

**截图：** 协商面板

---

### B6 — 批量确认阶段分工

1. 找到"最终分工确认"区域（深色背景面板）
2. 点击"确认当前阶段分工"按钮

**验证：**
- [ ] 该阶段所有已确认（owner_confirmed）的提案变为 finalized
- [ ] 已拒绝的提案不受影响
- [ ] 任务的 owner_user_id 正式更新
- [ ] 仪表盘"分工覆盖率"更新
- [ ] "推荐下一步"可能更新为"主动推进"

**截图：** 分工确认后状态

---

### B7 — 验证分工覆盖率

1. 查看仪表盘头部三个指标卡

**验证：**
- [ ] "分工覆盖率"百分比 = 有 owner 的任务数 / 总任务数 × 100%
- [ ] 数值与实际任务分配情况一致
- [ ] "待处理 P0" = P0 且非 done 的任务数
- [ ] "活跃行动卡" = status=active 的行动卡数

---

### B8 — 下一阶段分工

1. 如果项目有多个阶段，尝试为下一个阶段生成分工

**验证：**
- [ ] 新阶段可以独立生成分工推荐
- [ ] 新阶段的分工与上一阶段不冲突
- [ ] 不同阶段可以有不同负责人

---

### B9 — 全拒绝极限测试

1. 重置数据，重新加载种子数据
2. 运行分工推荐
3. 拒绝所有提案

**验证：**
- [ ] 所有提案状态变为"已拒绝"
- [ ] 页面不崩溃
- [ ] 可以重新运行"推荐分工"生成新提案
- [ ] 协商面板可能有多条协商记录

---

### B10 — 单成员场景

1. 重置数据
2. 创建只有 1 个成员的工作区和项目
3. 运行分工推荐

**验证：**
- [ ] Agent 仍能生成分工推荐（所有任务分配给同一人）
- [ ] 不崩溃、不报错
- [ ] reason 说明为什么全部分配给同一人

---

### B11 — 任务数多于成员数

1. 确认项目有 5 个任务但只有 3 个成员
2. 运行分工推荐

**验证：**
- [ ] 每个成员被分配多个任务
- [ ] 分工推荐覆盖所有任务
- [ ] reason 说明为什么一人负责多个任务

## 横向评估清单

### Agent 输出质量

- [ ] 分工是否考虑了成员技能（skill_match）
- [ ] 分工是否考虑了可用时间（availability_match）
- [ ] 分工是否考虑了角色偏好（preference_match）
- [ ] 分工是否尊重了成员限制（constraint_respected）
- [ ] reason 是否有具体依据（如"小王擅长 React，且有 15h/周可用时间"），而非泛泛而谈
- [ ] backup_owner 是否合理（有能力但非最优选的人）
- [ ] risk_note 是否有实际风险提示

### 协商质量

- [ ] 协商选项是否真正解决了拒绝原因
- [ ] Agent 是否编造了不存在的成员或任务
- [ ] agent_message 是否清晰说明协商建议

### 交互体验

- [ ] "接受分工"和"拒绝分工"按钮是否明确
- [ ] 拒绝表单展开动画是否流畅
- [ ] 分工状态变化是否有视觉反馈（Badge 颜色）
- [ ] "确认当前阶段分工"按钮位置是否合理
- [ ] 拒绝后偏好任务选择是否方便

### 状态一致性

- [ ] finalized 后任务 owner 才正式写入（owner_confirmed 时不写入）
- [ ] 拒绝后再确认是否正常
- [ ] 批量确认是否原子性（不会部分成功部分失败）
- [ ] 协商后原提案状态是否正确迁移
- [ ] 未 finalize 的分工不会修改 task.owner_user_id

### 数据完整性

- [ ] AssignmentProposal → AssignmentResponse 链路完整
- [ ] AssignmentResponse 包含 user_id / response / preferred_task_id / reason
- [ ] AssignmentNegotiation 包含 from_user_id / desired_task_id / agent_message
- [ ] 刷新页面后数据一致

### Agent 状态透明度

- [ ] 分工推荐成功时，显示绿色成功提示
- [ ] 分工推荐 fallback 时，提示明确标注"已使用基础建议"
- [ ] 分工推荐失败时，显示红色错误 + 重试入口
- [ ] 协商 Agent 输出同样遵循 success/repaired/fallback/failed 区分
