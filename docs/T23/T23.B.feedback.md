## 问题清单

【B-1/B-11】【现象】在 5 个任务、3 个成员的前置场景中，点击“推荐分工”后只生成 1 条提案，且没有 `backup_owner_user_id`。这也导致“任务数多于成员数”场景无法验证每个成员被分配多个任务、分工推荐覆盖所有任务。  
【期望】分工推荐应覆盖当前阶段待分配任务；任务数多于成员数时允许成员承担多个任务，并给出一人多任务的具体理由。  
【严重度】❌ 阻塞  
【截图】未单独截图；证据见 `output/t23b/test-data.json` 与浏览器分工流程面板。  

【B-1/B-2】【现象】mock fallback 分工理由为英文泛化模板，例如 `Fallback: 林舟 is the first available member. Must be confirmed.`；匹配信息也使用英文标签 `Skill` / `Availability` / `Preference` / `Constraint`。  
【期望】UI 标签与 fallback 文案应统一中文；reason 应结合成员技能、可用时间、偏好、限制和任务内容给出具体依据。  
【严重度】⚠️ 体验  
【截图】未单独截图；证据见分工提案详情。  

【B-2】【现象】首条提案缺少备选负责人，UI 显示“备选人：暂无备选”。在多成员场景中，fallback 未尝试挑选可作为 backup owner 的成员。  
【期望】多成员团队中应尽量生成合理备选人；如果没有备选人，应说明原因，例如技能不匹配或可用时间不足。  
【严重度】⚠️ 体验  
【截图】未单独截图；证据见分工提案详情。  

【B-3】【现象】接受分工后，提案状态变为“已确认”，任务 owner 仍为空，这是正确的；但仪表盘“分工覆盖率”仍为 `0%`。T23.B 同时要求 owner 不提前写入、覆盖率指标更新，当前指标只按 finalized task owner 计算，两者存在口径冲突。  
【期望】需要明确覆盖率口径：如果只统计 finalized owner，T23.B 文档应改为 finalized 后更新；如果要在 accepted 后更新，UI 应新增“已确认分工覆盖率”或类似指标，避免与实际 owner 覆盖率混淆。  
【严重度】⚠️ 体验  
【截图】未单独截图；证据见 B3 接受后的指标卡。  

【B-4】【现象】拒绝表单中“偏好任务”下拉包含任务列表，但选中任务后，选择框触发器显示原始 UUID，例如 `323d43a9-b0a0-4ff8-a919-56a002744899`，不是任务标题。  
【期望】选中后应继续显示任务标题，原始 ID 不应暴露给用户。  
【严重度】⚠️ 体验  
【截图】未单独截图；证据见拒绝表单展开状态。  

【B-5】【现象】协商记录能显示“来自 陈远”，但 `agent_message` 暴露原始 user/proposal/task UUID，例如“成员 5cfb... 拒绝分工 0841... 后，希望改做 323d...”。  
【期望】协商消息应使用成员名和任务名，清楚解释协商建议，不能把内部 ID 当正文展示。  
【严重度】❌ 阻塞  
【截图】未单独截图；证据见协商面板。  

【B-5】【现象】创建协商记录时，`current_owner_user_id` 被写成拒绝该提案的成员，而不是偏好任务当前 owner；当偏好任务尚未分配时，这个字段语义不准确。  
【期望】`current_owner_user_id` 应表示 desired task 当前负责人；若偏好任务未分配，应为空或使用明确的未分配状态。  
【严重度】⚠️ 体验  
【截图】无；证据见协商接口返回数据。  

【B-6】【现象】批量确认后，finalized 和 owner_rejected 提案仍然展示“接受分工”“拒绝分工”按钮。按钮没有根据终态隐藏或禁用，容易让用户重复操作。  
【期望】终态提案应隐藏操作按钮，或明确禁用并解释状态已结束。  
【严重度】⚠️ 体验  
【截图】未单独截图；证据见分工确认后状态。  

【B-8】【现象】切换到下一阶段 `Review Prep` 后再次运行“推荐分工”，新提案仍指向上一阶段已 finalized 的任务“设计数据库 Schema”，并产生重复提案。  
【期望】下一阶段应只针对当前 active stage 的未分配任务生成分工；不应对上一阶段已 finalized 的任务再次生成 proposed 提案。  
【严重度】❌ 阻塞  
【截图】无；证据见阶段切换后的提案接口结果。  

【B-9】【现象】全拒绝后可以重新运行“推荐分工”，页面不崩溃；但 rerun 仍为同一个已拒绝任务生成新的 proposed 提案，形成同任务重复提案，没有为其他任务或其他成员生成替代方案。  
【期望】重新推荐应避开已拒绝组合，优先覆盖其他未分配任务，或根据拒绝原因生成新的 owner/task 协商建议。  
【严重度】❌ 阻塞  
【截图】无；证据见 `output/t23b/b9-result.json`。  

【B-10】【现象】单成员、3 个任务场景下，Agent 只生成 1 条分工提案，未把所有任务分配给唯一成员；reason 仍是英文 fallback 模板，没有说明为什么全部由同一人承担。  
【期望】单成员场景应覆盖所有任务，backup 可为空，但 reason 应解释单人团队限制、任务优先级和工作量风险。  
【严重度】❌ 阻塞  
【截图】无；证据见 `output/t23b/b10-result.json`。  

【横向-状态一致性】【现象】分工 fallback 逻辑疑似始终选 `workspace_state.project.tasks[0]` 和第一个可用成员，未过滤 active stage、已 finalized 任务、已拒绝组合或 unassigned-only 任务。  
【期望】分工推荐输入应受当前阶段和任务状态约束；Agent 或 fallback 都不能编造成员/任务，也不能重复推荐已完成分工。  
【严重度】❌ 阻塞  
【截图】无；证据见 B1、B8、B9、B10 的重复现象。  

【横向-交互体验】【现象】重置数据或切换到新测试项目后，导航栏用户切换器可能显示“选择身份”。原因是 localStorage 中残留的 current user ID 已不属于当前 workspace，但项目页没有自动 fallback 到 `project.created_by` 或 workspace 首个成员。  
【期望】当本地 current user ID 不在当前 workspace 成员列表中时，应自动修正为有效成员，并同步导航栏展示。  
【严重度】⚠️ 体验  
【截图】未单独截图；证据见 B9/B10 页面状态。

【期望】自动化测试应完全隔离本机真实 `.env`；需要真实 provider 的诊断测试应单独标记，并由显式环境变量开启。
