# T23.A — 冷启动 + 规划流测试

## 你的任务

从零开始走完：注册 → 创建工作区 → 填写负责人资料 → 添加成员资料 → 新建项目 → Agent 方向澄清 → 阶段规划 → 任务拆解。

验证一个完全不了解项目的新用户，能否无文档走通整个流程，以及 Agent 输出质量是否达标。

---

## 环境准备

### 1. 启动后端

```bash
cd backend
.venv\Scripts\Activate.ps1
python -m uvicorn app.main:app --reload --port 8000
```

确认 `http://localhost:8000/docs` 可访问。

### 2. 启动前端

新开一个终端：

```bash
cd frontend
npm install   # 首次需要
npm run dev
```

确认 `http://localhost:3000` 可访问。

### 3. 确认 LLM 配置

打开 `backend/.env`，确认：

- **第一轮（mock 模式）**：`LLM_PROVIDER=mock`
- **第二轮（真实 LLM）**：`LLM_PROVIDER=openai`，`LLM_API_KEY=你的key`，`LLM_MODEL=gpt-4o-mini`

修改 `.env` 后后端会自动重载。

### 4. 清空旧数据

在浏览器访问 `http://localhost:8000/docs`，找到 `POST /api/seed/reset`，点击 "Try it out" → "Execute"。或用前端首页的"重置演示"按钮。

### 5. 关于多成员操作

导航栏右侧有**用户切换下拉菜单**，可以切换当前身份为工作区内任意成员。切换后，签到表单、行动卡、任务状态更新等自动跟随当前身份。

- 默认身份是项目创建者
- 切换身份后刷新页面，身份保持（localStorage 持久化）
- 为其他成员填写 profile 仍需通过 API（`POST /api/member-profiles`，body 传对应 `user_id`），因为前端 profile 表单只服务于当前用户

### A18 — Agent 失败 fallback（mock 模式特有）

在 mock 模式下 Agent 通常不会失败。如果你能触发真实 LLM 超时（如设置 `LLM_TIMEOUT_SECONDS=1` 后运行 Agent），观察：

**验证：**
- [ ] 超时后显示错误消息（红色提示）
- [ ] 页面不崩溃
- [ ] 可以重新运行 Agent 操作
- [ ] 如果 fallback 触发，输出有合法默认值（无 None/空字段）

---

### A19 — Agent 状态透明度

每次 Agent 运行后，检查 Agent 操作区域的状态反馈：

**验证：**
- [ ] Agent 成功时，显示绿色成功提示
- [ ] Agent 输出被修复（repaired）时，提示中标注"输出已修复"而非伪装成完全成功
- [ ] Agent 使用 fallback 时，提示中明确标注"已使用基础建议"——fallback 不应被当作 AI 成功
- [ ] Agent 失败时，显示红色错误提示，包含失败原因和重试入口
- [ ] 在 Agent 时间线中，每条事件显示正确的状态（success / repaired / fallback / failed）

## 横向评估清单

每个步骤都要关注以下维度，发现问题记录到反馈表。

### Agent 输出质量

- [ ] 方向卡内容是否针对项目想法（非泛泛而谈）
- [ ] 阶段划分是否合理（数量、时间、目标）
- [ ] 任务优先级是否恰当（P0 真的是最重要的吗）
- [ ] 每个输出是否都有 reason
- [ ] Agent 是否编造了不存在的成员/任务/阶段

### 交互体验

- [ ] 每步操作后是否有 loading 状态
- [ ] 确认/拒绝后页面是否自动更新
- [ ] 面包屑导航是否正确
- [ ] "推荐下一步"推断是否正确
- [ ] 中文是否统一（无英文混入）
- [ ] 表单是否使用 shadcn 组件

### 数据一致性

- [ ] 确认后刷新页面（F5），数据是否还在
- [ ] 方向卡字段是否完整（problem/users/value/deliverables/boundaries/risks/suggested_questions）
- [ ] 阶段字段是否完整（name/goal/start_date/end_date/deliverable/done_criteria）
- [ ] 任务字段是否完整（title/description/priority/estimated_hours/dependencies/acceptance_criteria/can_cut）
- [ ] 未确认的 Agent 提案不会静默修改项目状态

### Agent 状态透明度

- [ ] success / repaired / fallback / failed 四种状态在 UI 中有区分
- [ ] fallback 输出被明确标注为基础建议，不伪装成 AI 成功
- [ ] 失败时有重试入口
- [ ] 时间线中每条事件显示正确状态

### UI 状态覆盖

- [ ] 每个区域是否有 empty 状态（虚线边框 + 提示文字）
- [ ] Agent 运行中是否有 loading 动画
- [ ] Agent 失败是否有 error 提示
- [ ] Agent 成功是否有 success 提示
