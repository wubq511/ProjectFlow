# T23.B Assignment Flow Fix Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 修复 T23.B 反馈中分工推荐、拒绝协商、阶段切换、单成员/多任务极限场景和测试基线问题，让 B1-B11 能以 mock 模式稳定通过，并为真实 LLM 第二轮验证提供可靠基础。

**Architecture:** 以“当前 active stage 的可分配任务”为分工边界，把分工历史纳入 `WorkspaceState`，让 Agent prompt、fallback 和 schema validation 都能避开已 finalized、已确认、已拒绝组合与跨阶段任务。后端负责语义校验和协商消息生成，前端只做展示与用户操作，不再拼接业务文本或暴露原始 ID。

**Tech Stack:** FastAPI + SQLModel + Pydantic 后端，Next.js + React + TypeScript + shadcn/ui + Vitest 前端，SQLite 本地测试库。

| 优先级 | 范围 | 对应反馈 | 目标 |

| P0 | 协商数据与消息正确性 | B-5 | 协商消息不暴露 UUID，`current_owner_user_id` 语义正确 |

## 架构决策

1. **分工覆盖口径保持 finalized-only。** `task.owner_user_id` 只有 finalized 后写入；顶部“分工覆盖率”继续按有 owner 的任务数计算。B3 的“接受后覆盖率更新”改为在分工面板展示“已确认待定 X 条”，避免把未生效分工混入正式覆盖率。
2. **Agent 与 fallback 都必须基于 `WorkspaceState` 的分工历史决策。** 只靠 `tasks[0]` 和 `members[0]` 的 fallback 不满足 MVP 主动推进价值。
3. **后端生成协商业务文案。** 前端不再拼接 `agent_message`，避免 UUID 暴露和字段语义漂移。
4. **后端验证比前端按钮更重要。** 即使前端隐藏终态按钮，API 仍必须拒绝 finalized、owner_rejected、owner_confirmed 的重复响应。
5. **真实 LLM 输出不能只校验 ID 存在。** Pydantic reference validation 要加入 active stage、可分配任务、重复 task、backup 与 owner 不同、拒绝组合避让等语义规则。

- [ ] **Step 1: 增加 assignment helper**

在 `backend/app/agent/modules/common.py` 增加以下职责清晰的 helper：

```python
TERMINAL_ASSIGNMENT_STATUSES = {"finalized", "owner_confirmed", "proposed", "negotiating"}


def active_stage_id(workspace_state: WorkspaceStateResponse) -> str | None:
    if not workspace_state.project:
        return None
    if workspace_state.project.current_stage_id:
        return workspace_state.project.current_stage_id
    active = next((stage for stage in workspace_state.project.stages if stage.status == "active"), None)
    return active.id if active else None


def rejected_assignment_pairs(workspace_state: WorkspaceStateResponse) -> set[tuple[str, str]]:
    if not workspace_state.project:
        return set()
    return {
        (proposal.task_id, proposal.recommended_owner_user_id)
        for proposal in workspace_state.project.assignment_proposals
        if proposal.status == "owner_rejected"
    }


def blocked_assignment_task_ids(workspace_state: WorkspaceStateResponse) -> set[str]:
    if not workspace_state.project:
        return set()
    return {
        proposal.task_id
        for proposal in workspace_state.project.assignment_proposals
        if proposal.status in TERMINAL_ASSIGNMENT_STATUSES
    }


def assignable_tasks(workspace_state: WorkspaceStateResponse) -> list[TaskState]:
    stage_id = active_stage_id(workspace_state)
    if not workspace_state.project or not stage_id:
        return []
    blocked_task_ids = blocked_assignment_task_ids(workspace_state)
    return [
        task
        for task in workspace_state.project.tasks
        if task.stage_id == stage_id
        and task.status != "done"
        and not task.owner_user_id
        and task.id not in blocked_task_ids
    ]
```

成员评分 helper 采用可解释规则，不引入复杂模型：
- 任务标题/描述包含技能关键词：+3。
- role_preference 或 interests 命中任务关键词：+2。
- 可用时间大于等于任务 estimated_hours：+2，否则 +1。
- constraints 非空但未命中任务关键词：+1；命中冲突关键词：-2。
- 同一轮 fallback 已分配任务越多，得分按负载递减。
