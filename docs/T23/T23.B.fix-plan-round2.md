## IR1 — 技能名中文化映射（P0）

**现象：** 真实 LLM 生成的 `skill_match` 中包含 `ai_ml`、`prompt_engineering` 等英文下划线格式，与 UI 中文语境不一致（如 `skill_match: "小赵ai_ml等级4、prompt_engineering等级4"`）。

**根因：** `output_schemas.py` 的 `_validate_references` 只校验 ID 存在性，不做文本合法化。LLM 直接输出 `MemberState.skills` 中的 `name` 字段（数据库存储值），未经过中文映射。

**方案选择：** 有三层可以修——

1. **Agent prompt 层：** 在 `assignment_recommendation.py` 的 `user_prompt` 中增加指令："技能名称请使用中文（如 AI/ML、Prompt 工程、后端开发）"，依赖 LLM 自行翻译。代价：不可靠，不同 LLM 行为不一致。
2. **Seed 数据层：** 把 `skills` 的 `name` 从 `ai_ml` 改成 `AI/ML`。代价：破坏现有数据库和测试，且用户创建的新 member 仍可能有英文 skill name。
3. **输出后处理层（推荐）：** 在 `output_schemas.py` 的 `validate_agent_output` 返回前，或 `agent_flow_service.py` 持久化前，对 `skill_match`/`availability_match`/`preference_match` 等字段做 skill name 替换。

**推荐方案：Prompt + 后处理双保险。** Prompt 层引导 LLM 用中文，后处理做兜底替换。映射表可维护在后端常量中。

**影响文件：**
- Modify: `backend/app/agent/modules/assignment_recommendation.py` — prompt 增加技能中文化指令
- Modify: `backend/app/agent/output_schemas.py` — 新增 `_normalize_user_facing_text()` 对 assignment output 做 skill name 替换
- Modify: `backend/app/agent/modules/common.py` — 新增 `SKILL_NAME_CN_MAP` 常量
- Test: `backend/app/tests/test_agent_output_schemas.py` — 增加 skill name 中文化验证

**详细步骤：**

- [ ] **Step 1:** 在 `common.py` 增加映射表
  ```python
  SKILL_NAME_CN_MAP: dict[str, str] = {
      "ai_ml": "AI/ML",
      "prompt_engineering": "Prompt 工程",
      "backend": "后端开发",
      "frontend": "前端开发",
      "database": "数据库",
      "devops": "DevOps",
      "design": "UI 设计",
      "animation": "动效设计",
      "testing": "测试",
      "documentation": "文档",
      "project_management": "项目管理",
      "presentation": "演示汇报",
  }
  ```

- [ ] **Step 2:** 在 `output_schemas.py` 的 `validate_agent_output` 中，对 `AssignmentRecommendationOutput` 类型的 output，遍历 `assignments` 中的 `skill_match`、`availability_match`、`preference_match`、`constraint_respected`、`risk_note`、`reason`，做 `str.replace(eng_key, cn_value)` 全量替换。

- [ ] **Step 3:** 在 `assignment_recommendation.py` 的 prompt 末尾增加：
  ```
  技能名称请使用中文：后端开发、前端开发、AI/ML、Prompt 工程、数据库、DevOps、UI 设计、动效设计、测试、文档、项目管理、演示汇报。不要使用英文下划线格式如 ai_ml、prompt_engineering。
  ```

- [ ] **Step 4:** 在 `test_agent_output_schemas.py` 增加 `test_t23b_skill_names_normalized_to_chinese`。

**验证：** 重新跑 `/api/agent/assign`，检查 `skill_match` 内容不包含 `ai_ml` 或 `prompt_engineering` 等英文下划线。

## IR2 — GLM-5.1 JSON 输出严格性（P0）

**现象：** 真实 LLM 二轮 B1 结果为 `status=repaired`，GLM-5.1 返回的 JSON 被 `_parse_or_repair_json()` 处理后才能通过 `json.loads()`。

**根因：** GLM-5.1 对 `response_format: json_object` 支持不完整，可能在 JSON 前后包裹 markdown code block 或尾随逗号。当前 `_repair_json_text()` 已兜底，但 `repaired` 状态意味着每次都要多一次解析尝试 + 没有 retry（一次成功是因为 repair 生效）。

**方案：**

1. **降低 temperature：** 当前 `temperature=0.2`，降到 `0.05` 可减少输出随机性、提高 JSON 合规率。
2. **Prompt 强化：** 在 system prompt 或 user_prompt 末尾加 "Return ONLY a valid JSON object. No markdown code blocks, no trailing commas, no comments."
3. **增加 retry budget（已是 2 次）：** 当前 `generate_structured_output` 已做 2 次尝试，但 `repaired` 不算一次完整 retry——repair 成功后直接返回。如果 repair 失败才进入下一次。可考虑：第一次 raw 失败后，在第二次尝试前注入 explicit JSON 格式纠正 prompt。

**推荐方案：** 组合 temperature 降低 + prompt 强化。不改 retry 逻辑（当前逻辑完善）。

**影响文件：**
- Modify: `backend/app/agent/llm_client.py` — `temperature` 从 `0.2` 降到 `0.05`（仅对 `complete()` 方法）
- Modify: `backend/app/agent/prompts.py` — system prompt 增加 JSON 格式要求

**详细步骤：**

- [ ] **Step 1:** 在 `llm_client.py` 的 `OpenAICompatibleLLMClient.complete()` 中，`temperature` 从 `0.2` 改为 `0.05`。
- [ ] **Step 2:** 在 `prompts.py` 的 `build_prompt_messages()` 中，system prompt 末尾增加：
  ```
  IMPORTANT: Return ONLY a valid, parseable JSON object. Do NOT wrap in markdown code blocks (```json). No trailing commas. No comments. Just pure JSON.
  ```
- [ ] **Step 3:** 重新跑真实 LLM 的 `/api/agent/assign`，确认 `status=success`（非 `repaired`）。

**验证：** 真实 LLM 二轮复测 B1，确认 `status=success`。

## IR4 — B8 非 active 阶段分工触发（P1）

**现象：** `active_stage_id()` 只返回 `project.current_stage_id` 或第一个 status=active 的阶段。当用户想为下一个 pending 阶段预分工作分工时，没有 API 层面的机制。

**根因：** 分工推荐硬绑定 active stage，没有提供 stage_override 参数。

**方案：** 在 `AgentFlowRequest` 增加可选 `stage_id` 字段；在 `recommend_assignments()` 接受可选的 stage override。这允许前端传入任意阶段 ID（如 pending 的下一个阶段）来触发分工推荐。

**影响文件：**
- Modify: `backend/app/schemas/agent_flow.py` — `AgentFlowRequest` 增加 `stage_id: str | None = None`

- Modify: `backend/app/agent/modules/assignment_recommendation.py` — `build_request()` 接受 `stage_id` override
- Modify: `backend/app/agent/modules/common.py` — `assignable_tasks()` 接受 `stage_id` override
- Modify: `backend/app/services/agent_flow_service.py` — 传递 `stage_id`
- Modify: `backend/app/api/routes_agent.py` — 传递 `data.stage_id`
- Test: `backend/app/tests/test_agent_modules.py` — 增加 stage override 测试

**详细步骤：**

- [ ] **Step 1:** 在 `AgentFlowRequest` 增加 `stage_id: str | None = None`
- [ ] **Step 2:** 在 `common.py` 的 `assignable_tasks()` 签名增加 `stage_id: str | None = None`，有值时优先使用
- [ ] **Step 3:** 在 `assignment_recommendation.py` 的 `build_request()` 签名增加 `stage_id: str | None = None`
- [ ] **Step 4:** 在 `coordinator.py` 的 `recommend_assignments()` 签名增加 `stage_id: str | None = None`

- [ ] **Step 6:** 在 `routes_agent.py` 中从 `data.stage_id` 传递
- [ ] **Step 7:** 增加 `test_t23b_assignment_with_stage_override` 测试

## IR5 — 后端 pytest 隔离真实 `.env` key（P1）

**现象：** `test_missing_api_key_returns_error` 因 `conftest.py` 只 pop 环境变量，但 `Settings(env_file=".env")` 直接从文件读取，导致实际读取到本机真实 API key、诊断返回 `ok` 而非 `error`。

**根因：** Pydantic Settings 的 `env_file` 参数在文件存在时总会加载，不受 `os.environ` 控制。

**方案：** 在 `conftest.py` 中通过 monkeypatch 覆盖 `Settings.model_config` 的 `env_file` 为不存在的路径，或在 test 中显式创建不带 env_file 的 Settings 实例。

**推荐方案：** 在 `conftest.py` 的 session fixture 中设置 `os.environ["LLM_API_KEY"] = ""`，同时修改 `test_missing_api_key_returns_error` 使用 `LLMDiagnosticRequest(provider="openai")` 显式传入空 key 来测试。

**影响文件：**
- Modify: `backend/app/tests/conftest.py` — 强制 `LLM_API_KEY=""` 而非 pop
- Modify: `backend/app/tests/test_llm_provider.py` — 增加隔离断言

**详细步骤：**

- [ ] **Step 1:** 在 `conftest.py` 中把 `os.environ.pop("LLM_API_KEY", None)` 改为 `os.environ["LLM_API_KEY"] = ""`
- [ ] **Step 2:** 在 `test_llm_provider.py` 增加 `test_test_env_key_is_empty()` 断言
- [ ] **Step 3:** 运行 `pytest app/tests/test_llm_provider.py -q`，确认全部通过

**验证：** `test_missing_api_key_returns_error` 返回 `status=error`，所有 LLM provider 测试通过。
